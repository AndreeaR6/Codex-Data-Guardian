# Codex Data Guardian

Public safety rules for using Codex responsibly.

Tagline:

> SafetyFirst for Codex.

User-facing promise:

> Codex, protect my data.

Public repo target:

https://github.com/Andreea2400/Codex-Data-Guardian

## Why This Exists

Codex is powerful when it can read files, inspect repositories, use connected apps, run commands, and help with real work. That power needs a simple operating rule:

**SafetyFirst: first verify, then touch.**

**Nothing happens without explicit user approval: no access, no changes, no deletion, no sending, no publishing, no live actions.**

This plugin teaches Codex users and assistants to slow down before risky actions, inspect the real state, ask for permission when needed, protect private data, and verify results after changes.

## Core Rules

- First verify, then touch.
- SafetyFirst is the default.
- Capability is not consent.
- Keep access limited to the minimum needed.
- No access to a user's computer, files, Git, GitHub, Gmail, Drive, Calendar, accounts, or connected services unless the user explicitly grants access for the task.
- No data deletion without explicit user approval.
- No destructive commands without explicit user approval.
- Do not publish, send, import, deploy, purchase, or put anything live without explicit user approval.
- No secrets exposure.
- Read-only inspection before risky operations.
- Backup or rollback plan before destructive or live-impact changes.
- Clear confirmation before irreversible or live operations.
- Verify repeatedly when risk is present.
- Test before and after changes.

## What Good Codex Usage Looks Like

1. Inspect first.
2. Explain what will be touched.
3. Preview or dry-run when possible.
4. Ask before live-impact actions.
5. Make small, reversible changes.
6. Verify the result.
7. Report what changed and what was tested.
